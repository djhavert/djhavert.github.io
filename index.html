512 LiveView

Description

This project was coded 100% by me from scratch in C++. It is extremely efficient, capable of running analysis on 20 MB of data per second, and displaying the results of that analysis live in multiple graphical windows. My PhD research involves recording electrical signals from brain slices on a Multi-electrode array. We record from 512 electrodes at 20,000 Hz, culminating in a data rate of 1.2 GB per minute. This enormous amount of data needs to be displayed live to the user in an easily readable way. The purpose of this software is to provide an open-source application that quickly and clearly displays the data in easy to read visualizations.

We used a different software in the past, but found it's functionalilty to be inadequate for two reasons.

  • The first reason was that the graphical interface only updated once a second. My software on the other hand, updates its graphical interface every frame, providing a much clearer picture of where bursts of activity originate and travel to.
  • The other factor was the need for additional features as our research became more complicated. The old software was closed-source and abandoned. By creating a new software from scratch, I could add any new features our lab wanted and make the code open-source so future researchers could modify the code to their own needs. For example, the old software did not have any knowledge about stimulation pulses given to the electrode array. The new software is capable of removing stimulation artifacts and generating plots related to the stimulation-driven activity.

Front-End Features

  • Oscilliscope
    • Displays the signal from a single channel, updating in real time.
    • Change to neighboring channels with a click of a button, or input any channel number to instantly change the display.
    • Pause the screen on the current channel and zoom in to see spike waveforms.
  • Spike Summary
    • Shows location of spiking neurons over the electrode map.
    • The size of the dot represents the amplitude of the spike detected, and hence the distance from the electrode. Larger dots represent spikes occuring very close to an electrode.
    • This window is extremely useful in determining where a slice is most active, which in turn helps us decide which electrodes to observe in the oscilliscope window, and which electrodes would be the most effective sites for electrical stimulation.
  • Stimulation & Post Stimulus Time Histogram (PSTH)
    • Ability to supply a stimulation file and see the immediate effects of stimulation.
    • In the PSTH window, visualize spiking activity as a direct result of stimulation.
    • Use the live PSTH plots to plan which electrodes to choose for your next stimulation experiment.
    • This feature is new and extremely useful as it allows us to immediately see the effects of stimulation. Previously we had to wait until we analyzed the data on a separate more powerful computer to see these effects.

Back-End Features

Multithreaded: The application is split up to run three distinct loops main loops on three separate threads.

  • Client-Server Thread
    • The data enters the recording computer via a National Instruments chip running custom LabVIEW software. A server is created by this software and the raw data is compressed and sent out via network in arbitrarily sized chunks.
    • My application sets up both a client and server to communicate with the chip. The client continuously receives and unpacks the data sent from the chip. The server sends commands like 'start' and 'stop' back to the chip.
    • All data received from the electrode array is placed in a data buffer that grows and shrinks as needed.
  • Analysis Thread
    • The primary reason for choosing C++ for this application is that it is extremely fast and efficient.
    • The first hurdle in electrode array recordings of neurons is identifying spikes in the raw voltage signals. Spikes come in many different shapes, sizes, and durations, and identifying them can be extremely computationaly expensive.
    • We receive data from 512 channels at a rate of 20 MB/s. This data needs to be combed through in real time, matching raw data to typical spike waveforms and ultimately identifying when and where spikes occur.
    • Identified spikes are registered and displayed in the Spike Summary window as dots varying in size and opacity.
    • If a stimulation file was supplied, the application keeps track of when these events occur, and automatically filters out the massive artifacts caused by the stimulation pulses.
  • Graphics Thread
    • The application is being displayed to Windows through the use of GLFW and OpenGL.
    • The GUI is made using Dear ImGUI, an open-source library made for C++ that is extremely fast and efficient due to its implementation of the Immediate Mode GUI paradigm.
    • Performance is steady, but the user can easily minimized any unused windows to boost performance.
    • It is extremely easy for future researchers to add more features and windows to this application.

About Me

My name is Danny. I am currently a PhD student studying the effects of stimulation on the cortical region of rat brain slices. Specifically, my research lies in exploring the efficacy of using a brain slice as a reservoir computer in pattern recognition tasks. I have great passion for both Science and Programming, and I especially enjoy writing software to assist in scientific analysis.

My first programming class was Introduction to C, taken during the summer between my first 2 years as an undergrad. C is certaintly cumbersome, but I loved programming so much I took an Object Oriented Programming in C++ class a couple years later. That was when I knew I wanted to pursue a career in software one day. I still had a passion in science though, so I decided to go for my PhD in Physics.

In grad school I never stopped programming. When I joined the lab, we were recording hundreds of gigabytes of data a day, but the analysis of that data took very long and was extremely cumbersome. I created a pipeline for the many different and complicated analysis tools we use, then wrote the necessary scripts to run this pipeline on the university supercomputers. Now everyone in the lab uses my pipeline and the time needed to run analysis on our data has been reduced from weeks to hours. I also created 512LiveView, detailed above, which has helped to future proof our lab by replacing old deprecated software with an open-source alternative that already has extra features compared to the old software, and can be added on to by future members of the lab.

I love creating software that improves the workflow for others and contributes to the mission of the group I am part of. I hope to create a career out of my passion.
